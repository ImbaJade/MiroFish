## ⚡ 项目概述

**MiroFish** 是一款基于多智能体技术的新一代 AI 预测引擎。通过提取现实世界的种子信息（如突发新闻、政策草案、金融信号），自动构建出高保真的平行数字世界。在此空间内，成千上万个具备独立人格、长期记忆与行为逻辑的智能体进行自由交互与社会演化。你可透过「上帝视角」动态注入变量，精准推演未来走向——**让未来在数字沙盘中预演，助决策在百战模拟后胜出**。

> 你只需：上传种子材料（数据分析报告或者有趣的小说故事），并用自然语言描述预测需求</br>
> MiroFish 将返回：一份详尽的预测报告，以及一个可深度交互的高保真数字世界

### 我们的愿景

MiroFish 致力于打造映射现实的群体智能镜像，通过捕捉个体互动引发的群体涌现，突破传统预测的局限：

- **于宏观**：我们是决策者的预演实验室，让政策与公关在零风险中试错
- **于微观**：我们是个人用户的创意沙盘，无论是推演小说结局还是探索脑洞，皆可有趣、好玩、触手可及

从严肃预测到趣味仿真，我们让每一个如果都能看见结果，让预测万物成为可能。

## 🔄 工作流程

1. **图谱构建**：现实种子提取 & 个体与群体记忆注入 & GraphRAG构建
2. **环境搭建**：实体关系抽取 & 人设生成 & 环境配置Agent注入仿真参数
3. **开始模拟**：双平台并行模拟 & 自动解析预测需求 & 动态更新时序记忆
4. **报告生成**：ReportAgent拥有丰富的工具集与模拟后环境进行深度交互
5. **深度互动**：与模拟世界中的任意一位进行对话 & 与ReportAgent进行对话


## 📴 纯离线部署（开源组件替代联网依赖）

从当前版本开始，后端支持离线图记忆后端：

- `MEMORY_BACKEND=local`：使用项目内置本地 JSON 图存储与混合检索（不依赖 Zep Cloud）。
- `MEMORY_BACKEND=mem0`：在 local 图谱上叠加 mem0 语义记忆检索（更接近 Zep 检索体验）。
- `OFFLINE_MODE=true`：放宽云端密钥校验，适配内网/离线环境。

推荐搭配本地已部署的 OpenAI 兼容 LLM 服务（如 vLLM）：

```env
OFFLINE_MODE=true
MEMORY_BACKEND=mem0
LLM_BASE_URL=http://host.docker.internal:8000/v1
LLM_MODEL_NAME=Qwen/Qwen2.5-14B-Instruct
# OpenAI 兼容接口可使用任意占位 key
LLM_API_KEY=offline-local-key
```

如使用 Docker Compose，可直接启动应用并连接你本地已部署的模型服务：

```bash
docker compose up -d
```

> 说明：`MEMORY_BACKEND=zep` 仍可继续使用原有云端图谱流程；`mem0` 未安装时会自动降级到 `local`。

## 🚀 快速开始

### 一、源码部署（推荐）

#### 前置要求

| 工具 | 版本要求 | 说明 | 安装检查 |
|------|---------|------|---------|
| **Node.js** | 18+ | 前端运行环境，包含 npm | `node -v` |
| **Python** | ≥3.11, ≤3.12 | 后端运行环境 | `python --version` |
| **uv** | 最新版 | Python 包管理器 | `uv --version` |

#### 1. 配置环境变量

```bash
# 复制示例配置文件
cp .env.example .env

# 编辑 .env 文件，填入必要的 API 密钥
```

**必需的环境变量：**

```env
# LLM API配置（支持 OpenAI SDK 格式的任意 LLM API）
# 推荐使用阿里百炼平台qwen-plus模型：https://bailian.console.aliyun.com/
# 注意消耗较大，可先进行小于40轮的模拟尝试
LLM_API_KEY=your_api_key
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL_NAME=qwen-plus

# Zep Cloud 配置
# 每月免费额度即可支撑简单使用：https://app.getzep.com/
ZEP_API_KEY=your_zep_api_key
```

#### 2. 安装依赖

```bash
# 一键安装所有依赖（根目录 + 前端 + 后端）
npm run setup:all
```

或者分步安装：

```bash
# 安装 Node 依赖（根目录 + 前端）
npm run setup

# 安装 Python 依赖（后端，自动创建虚拟环境）
npm run setup:backend
```

#### 3. 启动服务

```bash
# 同时启动前后端（在项目根目录执行）
npm run dev
```

**服务地址：**
- 前端：`http://localhost:3000`
- 后端 API：`http://localhost:5001`

**单独启动：**

```bash
npm run backend   # 仅启动后端
npm run frontend  # 仅启动前端
```

### 二、Docker 部署

```bash
# 1. 配置环境变量（同源码部署）
cp .env.example .env

# 2. 拉取镜像并启动
docker compose up -d
```

默认会读取根目录下的 `.env`，并映射端口 `3000（前端）/5001（后端）`

> 在 `docker-compose.yml` 中已通过注释提供加速镜像地址，可按需替换


## 📘 详细使用教程（从 0 到 1）

本章节提供一个完整的“首次使用路径”，按步骤走即可跑通一次完整预测流程。

### 1）启动后先做什么

1. 打开前端：`http://localhost:3000`
2. 如果启用了登录页，请使用演示账号登录（若你已改为自己的鉴权方式则按你的配置）
3. 回到首页后，按页面提示完成以下输入：
   - 上传种子文件（`pdf/md/txt`）
   - 输入模拟需求（自然语言）
4. 点击“启动引擎”创建项目

> 建议第一次先用**短文本 + 20~40轮以内**测试，便于快速验证端到端流程。

---

### 2）一次完整流程怎么走（对应 5 个页面）

#### Step 1：图谱构建
- 系统会先把上传文本切块并写入图记忆后端。
- 若 `MEMORY_BACKEND=zep`：使用 Zep 云图谱。
- 若 `MEMORY_BACKEND=local/mem0`：使用本地图存储（mem0 模式会额外做语义记忆增强）。

你重点关注：
- 是否成功创建 `graph_id`
- 节点数/边数是否大于 0
- 是否有构建异常日志

#### Step 2：环境搭建
- 从图谱读取实体，过滤为可用于模拟的实体类型。
- 生成 OASIS 所需 agent profile 与仿真配置。

你重点关注：
- 过滤后实体数量是否合理
- profile 文件是否成功生成

#### Step 3：开始模拟
- 系统会并行运行模拟世界（平台行为由 OASIS 驱动）。
- 可在前端观察进度与回合变化。

你重点关注：
- round 是否持续推进
- 是否出现中断/超时

#### Step 4：报告生成
- ReportAgent 调用检索工具与模拟结果生成报告。
- 在 `local/mem0` 下依然可用，但语义检索能力取决于本地后端质量。

你重点关注：
- 报告是否生成成功
- 报告中是否有关键证据链

#### Step 5：深度互动
- 与模拟角色或报告代理继续对话追问。

你重点关注：
- 回答是否引用了前序模拟事实
- 连续追问时是否保持上下文一致

---

### 3）三种图记忆后端怎么选

| 后端 | 配置 | 适用场景 | 特点 |
|---|---|---|---|
| `zep` | `MEMORY_BACKEND=zep` | 在线生产 / 高质量图检索 | 能力最强，但需要外网与 ZEP_API_KEY |
| `local` | `MEMORY_BACKEND=local` | 纯离线、快速部署 | 无外网依赖，功能完整可跑通 |
| `mem0` | `MEMORY_BACKEND=mem0` | 离线且希望更好语义召回 | local + mem0 语义增强；未安装会降级 local |

推荐：
- 本地 PoC：`local`
- 离线增强：`mem0`
- 正式高质量：`zep`

### 6）常见问题（FAQ）

#### Q1：页面一直在“初始化中”
- 先看后端控制台是否报错。
- 核对 `.env` 中 LLM 配置是否正确（`LLM_BASE_URL`、`LLM_MODEL_NAME`）。
- 离线模式请确认本地 LLM 服务端口可达（如 vLLM 的 `8000`）。

#### Q2：离线模式下检索质量不稳定
- 优先切换 `MEMORY_BACKEND=mem0`。
- 提升种子文本质量：结构化描述、减少噪音文本、分主题上传。
- 先做小规模回合验证，再逐步加大模拟轮数。

#### Q3：`MEMORY_BACKEND=mem0` 但日志提示回退 local
- 说明环境未安装 `mem0` 或初始化失败。
- 补装依赖后重启服务；若仍失败，可先用 `local` 保证流程可用。

#### Q4：报告质量不如预期
- 提示词尽量具体：对象、时间、变量、预期输出格式。
- 把需求拆分为多个子问题分批运行，再进行综合报告。

### 8）本地 vLLM / OCR / Re-Embedding / Rerank 变量配置

> 下面是“统一 API 网关式”写法：即使当前版本部分变量暂未在后端强制消费，也建议提前规范化，便于后续升级与多模型编排。

#### 推荐 `.env` 模板（可直接复制）

```env
# =========================
# 基础运行
# =========================
OFFLINE_MODE=true
MEMORY_BACKEND=mem0
FLASK_DEBUG=true

# =========================
# 主 LLM（OpenAI 兼容）- 推荐 vLLM
# =========================
LLM_API_KEY=offline-local-key
LLM_BASE_URL=http://127.0.0.1:8000/v1
LLM_MODEL_NAME=Qwen/Qwen2.5-14B-Instruct

# 可选：当你希望把不同任务路由到不同模型时
# VLLM_CHAT_BASE_URL=http://127.0.0.1:8000/v1
# VLLM_CHAT_MODEL=Qwen/Qwen2.5-14B-Instruct
# VLLM_REASONING_BASE_URL=http://127.0.0.1:8001/v1
# VLLM_REASONING_MODEL=deepseek-ai/DeepSeek-R1-Distill-Qwen-14B

# =========================
# OCR 服务（例如 PaddleOCR / RapidOCR 自建 API）
# =========================
OCR_API_BASE_URL=http://127.0.0.1:9001
OCR_API_KEY=
OCR_PROVIDER=local
OCR_TIMEOUT_SECONDS=60

# =========================
# Re-Embedding（向量化）服务
# =========================
EMBEDDING_API_BASE_URL=http://127.0.0.1:9002/v1
EMBEDDING_API_KEY=
EMBEDDING_MODEL_NAME=bge-m3
EMBEDDING_DIM=1024

# =========================
# Rerank 服务
# =========================
RERANK_API_BASE_URL=http://127.0.0.1:9003/v1
RERANK_API_KEY=
RERANK_MODEL_NAME=bge-reranker-v2-m3
RERANK_TOP_K=20

# =========================
# 图记忆（在线时用 zep；离线可忽略）
# =========================
ZEP_API_KEY=
```

#### vLLM 启动示例

```bash
python -m vllm.entrypoints.openai.api_server \
  --host 0.0.0.0 \
  --port 8000 \
  --model Qwen/Qwen2.5-14B-Instruct \
  --dtype auto \
  --max-model-len 32768
```

#### API 连通性自检（建议部署后立即执行）

```bash
# 1) LLM
curl http://127.0.0.1:8000/v1/models

# 2) OCR（按你的服务协议调整）
curl http://127.0.0.1:9001/health

# 3) Embedding（按你的服务协议调整）
curl http://127.0.0.1:9002/v1/models

# 4) Rerank（按你的服务协议调整）
curl http://127.0.0.1:9003/v1/models
```

#### 变量设计建议

- 统一所有模型服务为 OpenAI 兼容风格接口，便于切换实现（vLLM / Xinference / 自建网关）。
- OCR、Embedding、Rerank 建议都提供 `BASE_URL + API_KEY + MODEL_NAME` 三件套。
- 在生产环境建议把 `.env` 拆分为：
  - `.env.runtime`（运行参数）
  - `.env.secrets`（密钥）
  - 通过部署系统注入，不直接落盘。
